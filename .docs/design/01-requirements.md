# 01. Requirements — 유저 시나리오 기반 기능 정의 및 요구사항 명세

---

## 범위

회원가입 / 내 정보 조회 / 비밀번호 변경은 이전에 완료. 아래부터 다룬다.

---

## 유저 시나리오

> 시나리오 1~3(상품·브랜드 탐색)은 비회원도 접근 가능하다.
> 시나리오 4~6(좋아요·주문)은 로그인 완료 상태를 전제로 한다.

---

### 👤 고객

#### 시나리오 1. 고객이 초기 화면에서 상품을 탐색한다

고객이 서비스에 처음 진입하면 상품 목록이 바로 보인다.

| METHOD | URI | 인증 | 설명 |
|--------|-----|:----:|------|
| GET | `/api/v1/products` | X | 상품 목록 조회 |

**정렬**
- 기본 정렬은 최신순(`latest`)
- 고객은 정렬 기준을 변경할 수 있다: 최신순(`latest`) / 좋아요 많은 순(`likes_desc`) / 가격 낮은 순(`price_asc`)
- `likes_desc` 정렬에서 좋아요 수가 같은 경우 최신순으로 정렬한다
- 정렬 기준을 변경하면 목록이 즉시 갱신된다

**필터**
- 특정 브랜드를 선택하면 해당 브랜드의 상품만 표시된다
- 브랜드 필터와 정렬 기준은 함께 적용될 수 있다

**목록 항목**
- 각 상품 카드에는 상품명, 가격, 브랜드명, 좋아요 수가 표시된다
- 삭제된 상품은 목록에 노출되지 않는다
- 재고가 0인 상품은 목록에 노출된다

**페이지네이션**
- 한 페이지에 기본 20개 노출
- 더 많은 상품을 보려면 다음 페이지로 이동할 수 있다

---

#### 시나리오 2. 고객이 상품 상세를 조회한다

목록에서 상품을 선택하면 상세 정보를 볼 수 있다.

| METHOD | URI | 인증 | 설명 |
|--------|-----|:----:|------|
| GET | `/api/v1/products/{productId}` | X | 상품 상세 조회 |

**표시 정보**
- 상품명, 가격, 재고, 설명, 브랜드명, 좋아요 수

**예외**
- 존재하지 않거나 삭제된 상품 접근 → 404

---

#### 시나리오 3. 고객이 브랜드 정보를 조회한다

상품 상세에서 브랜드명을 클릭하면 해당 브랜드 정보를 볼 수 있다.

| METHOD | URI | 인증 | 설명 |
|--------|-----|:----:|------|
| GET | `/api/v1/brands/{brandId}` | X | 브랜드 정보 조회 |

**표시 정보**
- 브랜드명, 브랜드 설명 등 브랜드가 가진 기본 정보

**예외**
- 존재하지 않거나 삭제된 브랜드 접근 → 404

---

#### 시나리오 4. 고객이 상품에 좋아요를 누른다

| METHOD | URI | 인증 | 설명 |
|--------|-----|:----:|------|
| POST | `/api/v1/products/{productId}/likes` | O | 좋아요 등록 |
| DELETE | `/api/v1/products/{productId}/likes` | O | 좋아요 취소 |
| GET | `/api/v1/users/{userId}/likes` | O | 내 좋아요 목록 조회 |

**등록**
- 상품 상세에서 좋아요를 누르면 즉각 반영된다
- 같은 상품에 좋아요를 두 번 누를 수 없다 (중복 불가)

**취소**
- 이미 누른 좋아요를 취소할 수 있다
- 좋아요하지 않은 상품을 취소하려 하면 허용하지 않는다

**내 좋아요 목록 조회**
- 내가 좋아요한 상품 목록을 볼 수 있다

**예외**
- 존재하지 않는 상품에 좋아요 시도 → 404
- 이미 좋아요한 상품에 재등록 시도 → 409
- 좋아요하지 않은 상품 취소 시도 → 404

---

#### 시나리오 5. 고객이 주문을 요청하고 결제한다

로그인한 고객이 여러 상품을 한 번에 주문하고 결제까지 완료한다.

| METHOD | URI | 인증 | 설명 |
|--------|-----|:----:|------|
| POST | `/api/v1/orders` | O | 주문 생성 |
| POST | `/api/v1/orders/{orderId}/pay/start` | O | 결제 진입 (재고 선점) |
| POST | `/api/v1/orders/{orderId}/pay/confirm` | O | 결제 완료 (재고 확정) |

**주문 생성**
- 상품과 수량을 지정해 주문을 생성한다
- `couponId`를 함께 전달하면 해당 발급 쿠폰이 적용된다 (생략 가능)
- 쿠폰을 적용하는 경우 주문 1건당 1장만 사용할 수 있다
- 생성 직후 상태는 `PENDING_PAYMENT`이며 재고는 아직 차감되지 않는다
- 주문 정보에는 당시의 상품명·가격, 쿠폰 적용 전 금액, 할인 금액, 최종 결제 금액이 스냅샷으로 기록된다
  (이후 상품 정보나 쿠폰 정보가 바뀌어도 주문 기록은 변하지 않는다)

**결제 진입**
- 결제 화면 진입 시점에 재고를 선점한다 (`reservedStock` 증가)
- 재고가 부족하면 결제 진입 자체가 실패한다

**결제 완료**
- 결제가 완료되면 재고를 확정 차감한다 (`totalStock`, `reservedStock` 감소)
- 주문 상태가 `CONFIRMED`로 전환된다

**예외**
- 주문 항목이 비어있으면 주문할 수 없다 → 400
- 존재하지 않는 상품이 포함된 경우 주문 전체가 실패한다 → 404
- 존재하지 않거나 이미 사용된 쿠폰, 만료된 쿠폰, 타 유저 소유 쿠폰으로 요청 시 실패한다 → 400/403/404
- 결제 진입 시 재고가 부족한 상품이 있으면 실패한다 → 400

---

#### 시나리오 6. 고객이 주문 내역을 조회한다

로그인한 고객이 자신의 주문 이력을 확인할 수 있다.

| METHOD | URI | 인증 | 설명 |
|--------|-----|:----:|------|
| GET | `/api/v1/orders?startAt=2026-01-31&endAt=2026-02-10` | O | 주문 목록 조회 |
| GET | `/api/v1/orders/{orderId}` | O | 주문 상세 조회 |

**목록 조회**
- 조회 기간(시작일 ~ 종료일)을 지정해 주문 목록을 볼 수 있다
- 본인의 주문만 조회된다

**상세 조회**
- 단일 주문의 상세 내역(주문 항목, 수량, 당시 가격 등)을 볼 수 있다
- 본인의 주문만 상세 조회할 수 있다

**예외**
- 타인의 주문 상세 접근 → 403
- 존재하지 않는 주문 접근 → 404

---

#### 시나리오 9. 고객이 쿠폰을 발급받고 조회한다

| METHOD | URI | 인증 | 설명 |
|--------|-----|:----:|------|
| POST | `/api/v1/coupons/{couponId}/issue` | O | 쿠폰 발급 요청 |
| GET | `/api/v1/users/me/coupons` | O | 내 쿠폰 목록 조회 |

**쿠폰 발급**
- 어드민이 등록한 쿠폰 템플릿을 기반으로 발급받는다
- 동일한 쿠폰 템플릿을 중복 발급받을 수 없다
- 존재하지 않거나 삭제된 쿠폰 템플릿에는 발급이 불가하다
- 만료된 쿠폰 템플릿에는 발급이 불가하다

**내 쿠폰 목록 조회**
- 본인이 발급받은 쿠폰 목록을 최신 발급순으로 반환한다
- 각 쿠폰의 상태(`AVAILABLE` / `USED` / `EXPIRED`)를 함께 반환한다
  - `AVAILABLE`: 사용 가능
  - `USED`: 이미 사용 완료
  - `EXPIRED`: 템플릿의 만료 일시(`expiredAt`)가 현재 시각 이전

**예외**
- 존재하지 않는 쿠폰 템플릿 → 404
- 이미 발급받은 쿠폰 재발급 시도 → 409
- 만료된 쿠폰 템플릿 발급 시도 → 400

### 🔧 어드민

#### 시나리오 7. 어드민이 브랜드를 등록·관리한다

| METHOD | URI | 설명 |
|--------|-----|------|
| POST | `/api/v1/admin/brands` | 브랜드 등록 |
| PATCH | `/api/v1/admin/brands/{brandId}` | 브랜드 수정 |
| DELETE | `/api/v1/admin/brands/{brandId}` | 브랜드 삭제 |

**등록**
- 브랜드명은 필수 입력값이다

**수정**
- 이미 등록된 브랜드의 정보를 수정할 수 있다

**삭제**
- 브랜드를 삭제하면 해당 브랜드에 속한 상품도 함께 삭제된다
- 삭제된 브랜드와 상품은 고객 화면에서 더 이상 노출되지 않는다

**예외**
- 존재하지 않는 브랜드 접근 → 404

---

#### 시나리오 8. 어드민이 상품을 등록·관리한다

| METHOD | URI | 설명 |
|--------|-----|------|
| POST | `/api/v1/admin/products` | 상품 등록 |
| PATCH | `/api/v1/admin/products/{productId}` | 상품 수정 |
| DELETE | `/api/v1/admin/products/{productId}` | 상품 삭제 |

**등록**
- 브랜드는 선택 사항이다. 브랜드 없이 상품을 등록할 수 있다.
- 브랜드를 지정하는 경우, 기존에 등록된 브랜드여야 한다.
- 상품명, 가격, 재고, 설명은 필수 입력값이다
- 가격과 재고는 0 이상이어야 한다

**수정**
- 상품명, 가격, 설명은 수정 가능하다
- 상품의 브랜드는 변경할 수 없다 (브랜드 이동 불가)

**삭제**
- 단건 삭제 가능
- 브랜드 삭제 시 해당 브랜드의 상품은 자동으로 삭제된다

**예외**
- 존재하지 않는 브랜드 ID로 상품 등록 시도 → 404 (brandId가 제공된 경우에만 검증)
- 존재하지 않는 상품 접근 → 404

---

#### 시나리오 10. 어드민이 쿠폰을 등록·관리한다

| METHOD | URI | 설명 |
|--------|-----|------|
| POST | `/api-admin/v1/coupons` | 쿠폰 템플릿 등록 |
| GET | `/api-admin/v1/coupons?page=0&size=20` | 쿠폰 템플릿 목록 조회 |
| GET | `/api-admin/v1/coupons/{couponId}` | 쿠폰 템플릿 상세 조회 |
| PUT | `/api-admin/v1/coupons/{couponId}` | 쿠폰 템플릿 수정 |
| DELETE | `/api-admin/v1/coupons/{couponId}` | 쿠폰 템플릿 삭제 |
| GET | `/api-admin/v1/coupons/{couponId}/issues?page=0&size=20` | 특정 쿠폰의 발급 내역 조회 |

**등록**
- 쿠폰 타입은 정액(`FIXED`) 또는 정률(`RATE`) 중 하나다
- `FIXED`: `value`는 할인 금액(원), 1 이상이어야 한다
- `RATE`: `value`는 할인율(%), 1 이상 100 이하이어야 한다
- `minOrderAmount`는 선택값이며, 지정 시 최소 주문 금액 미충족이면 쿠폰 적용 불가
- `expiredAt`은 쿠폰 만료 일시 (필수)

**수정**
- `name`, `value`, `minOrderAmount`, `expiredAt`은 수정 가능하다
- `type`은 변경할 수 없다 — 이미 발급된 쿠폰의 할인 방식이 바뀌는 혼란 방지

**삭제**
- 소프트 딜리트로 처리한다
- 삭제 후에도 이미 발급된 쿠폰은 만료 전까지 정상 사용 가능하다

**예외**
- 존재하지 않는 쿠폰 템플릿 접근 → 404
- `type` 변경 시도 → 400

## 미결 사항 (비즈니스 규칙)

| # | 질문 | 결정 |
|---|------|------|
| 1 | 좋아요 수가 같은 상품이 여럿일 때 `likes_desc` 정렬의 2차 기준은? | 최신순 |
| 2 | 상품 목록의 각 카드에 브랜드명이 바로 보여야 하는가? | 바로 표시 |
| 3 | 재고가 0인 상품도 목록에 노출되는가? | 노출 |
| 4 | 한 상품이라도 결제 진입 시 재고 부족이면 결제 전체가 실패하는가? | 전체 실패 |
| 5 | `RATE` 쿠폰 할인 계산 시 소수점 처리 방식은? | 버림(floor) |
| 6 | 쿠폰 수정 시 `type` 변경 허용 여부는? | 불허 — 발급된 쿠폰의 기대 불일치 방지 |
| 7 | 쿠폰 템플릿 삭제 후 이미 발급된 쿠폰의 사용 가능 여부는? | 만료 전까지 사용 가능 유지 |
| 8 | 내 쿠폰 목록 정렬 기준은? | 최신 발급순 |

## 기능적 요구사항 (Functional Requirements)

**고객**
- **상품 목록 조회**: 최신순 / 좋아요순 / 가격 낮은 순 정렬, 브랜드 필터, 페이지네이션
- **상품 상세 조회**: 상품 정보 조회
- **브랜드 정보 조회**
- **좋아요 등록 / 취소**
- **내 좋아요 목록 조회**
- **쿠폰 발급**: 쿠폰 템플릿 기반 발급, 중복 발급 불가
- **내 쿠폰 목록 조회**: AVAILABLE / USED / EXPIRED 상태 포함
- **주문 생성**: 여러 상품을 한 번에, 쿠폰 적용 가능, 금액 스냅샷 저장
- **결제 진입**: 재고 선점
- **결제 완료**: 재고 확정 차감
- **주문 목록 조회**: 날짜 범위 필터
- **주문 상세 조회**

**어드민**
- **브랜드 관리**: 등록 / 수정 / 삭제 (삭제 시 소속 상품 함께 삭제)
- **상품 관리**: 등록 / 수정 / 삭제
- **쿠폰 관리**: 템플릿 등록 / 수정 / 삭제, 발급 내역 조회

---

## 비기능적 요구사항 (Non-Functional Requirements)

- **응답 속도**: 상품 목록 조회, 좋아요 등록 등 고객 인터랙션은 체감상 즉각 반영되어야 한다
- **데이터 불변성**: 주문 완료 후 상품·쿠폰 정보가 변경되어도 주문 기록의 스냅샷은 변하지 않아야 한다
- **데이터 일관성**: 주문 생성 시 쿠폰 사용·재고 선점·주문 저장은 하나의 트랜잭션으로 원자적으로 처리되어야 한다
- **동시성 제어**: 동일 쿠폰 동시 사용, 동일 재고 동시 차감 시 Lost Update가 발생하지 않아야 한다
- **접근 제어**: 고객은 본인의 데이터(주문, 쿠폰)만 접근할 수 있다
